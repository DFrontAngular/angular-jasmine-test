Control de acceso real con HTTP, guards funcionales y testing moderno.

Guard funcional con lógica asíncrona

En Angular moderno los guards pueden ser funciones que usan inject() directamente. En este caso validamos el rol del usuario realizando una petición HTTP a /api/me.

El guard devuelve un Observable<GuardResult>, que puede ser true o un UrlTree de redirección. 

{{ guardCode }}
Un guard funcional debe ejecutarse dentro de un contexto de inyección, ya que utiliza inject().

Testing del guard con HTTP mockeado

Como el guard realiza una petición HTTP, necesitamos HttpTestingController para interceptarla.

En lugar de usar subscribe() y done(), convertimos el Observable en Promise usando firstValueFrom y trabajamos con async/await. 

{{ guardTestCode }}
Siempre que se use HttpTestingController es obligatorio hacer flush() o error() para evitar tests colgados.

Ejecución en contexto de inyección

Los guards funcionales que utilizan inject() deben ejecutarse dentro de TestBed.runInInjectionContext().

Esto garantiza que Angular pueda resolver correctamente las dependencias como HttpClient o Router.

Test de navegación real

Para probar navegación no testeamos el Router internamente. Utilizamos RouterTestingHarness y verificamos lo que realmente se renderiza. 

{{ routingTestCode }}
Los tests de routing deben validar comportamiento observable en el DOM, no implementación interna del router.

Buenas prácticas aplicadas

  • ✔ Guards funcionales en lugar de clases
  • ✔ Uso de inject() en vez de constructor
  • ✔ HTTP completamente mockeado
  • ✔ Sin any en tests (strict mode)
  • ✔ Sin done() ni subscribe() manual
  • ✔ Uso de async/await + firstValueFrom
  • ✔ Verificación de peticiones con httpMock.verify()
El objetivo no es solo que el guard funcione, sino que sea testeable, mantenible y tipado estrictamente.
