Cómo demasiados if afectan directamente a tu cobertura y a la calidad del diseño.

¿Qué mide realmente el Coverage?

{{ coverageConcept }}

El problema real suele estar en Branches. Cada if crea al menos dos caminos posibles.

Más if → Más ramas → Más tests necesarios.

Impacto real en Branch Coverage

Un if simple genera 2 ramas. Un if dentro de otro genera combinaciones.

El número de caminos crece exponencialmente.

Más complejidad ciclomática → Más tests → Más deuda técnica.

Refactor usando Record y simplificación

{{ refactoredExample }}

Eliminamos múltiples if y reducimos ramas. Ahora solo hay una condición especial.

Menos ramas → Menos tests → Mejor mantenibilidad.

Conclusión

  • El coverage no es solo porcentaje.
  • La complejidad ciclomática impacta directamente.
  • Muchos if indican posible problema de diseño.
  • Refactorizar mejora arquitectura y testing.
Código fácil de testear suele ser código bien diseñado.
